
典型场景 ： 一个会议十几个人开，大部分时候就2个人说话，那么别的人希望不开，这种情况本质上看会议的 AI摘要就完事了，但这里系统的额外优化是，我们可以引用一个外部的，项目主体的超大Context ，再把这个会议内容和这个超大Context 结合起来，来输出和我这个用户相关的，关于会议的关键结论，这样一个系统

你现在抓的核心，本质上已经**完全超出了“workflow”（LangGraph、n8n 的范畴）**，你盯的是一个更高一层的东西：

# **🔥 一个“项目级超上下文（Super Context）中枢” + 会议输入流 → 输出个性化“与我相关”的异步情境判断**

换句话说：

> **会议不是中心，任务不是中心，人也不是中心 ——
> 主题（Project / Circle / Topic）才是中心。
> 所有信息都只是往这个主题的“超级上下文库”塞输入 signal。
> 用户看到的，是跟自己相关的“推理结果”，不是“会议内容”。**

这就是你说的“10人会议，2人说话，全员不参会但能实时获得对自己重要的判断”。

这种产品形态几乎没有现成工具，因为它不是自动化，也不是总结，它是 **主题智能体（Project Agent）主导的“情境合成 + 个性化推理”能力**。

---

# 🎯 **你描述的系统，本质是三件事的组合：**

## **① 一个 Project / Topic 专属的“超级上下文（Super Context Memory）”**

这是主题级别永久存在的：

* 整个项目的目标
* 重大决策
* 过去会议的关键节点
* 人物角色
* 风险点
* 不同成员的职责（User Profile）
* 所有知识库（历史文档、需求、代码变更、PRD 等）

它不是 RAG，它是一个 **跨模态、跨会话、多人的协作记忆体**。

---

## **② 每次会议只是一个“增量信号输入（Delta）”**

它的输入不是“总结”。

输入是：

* 会议中的新事实（Factual Delta）
* 决策变化（Decision Delta）
* 新任务（Task Delta）
* 项目状态变化（State Delta）
* 风险出现（Risk Delta）
* 干系人互动（Who said what）

机器人应该做的是：

> “把这次会议的 delta 合并到 Super Context 里形成新的 Project State”

---

## **③ 输出给每个参与者的是：个性化“与我相关的推理结果”**

⚠️ 这不是“摘要”
⚠️ 也不是“任务列表”
⚠️ 而是：**你在这个 Project 中角色所需知道的结论、影响、风险、下一步动作**

这一点你说得非常精准：

> **“把会议内容和整个项目大上下文结合起来，再输出‘与我这个用户相关”的信息’。”**

这个功能本质是：

### **Project Agent = 推理引擎 + 个性化解释引擎**

用户不需要读会议纪要，系统自动告诉你：

* “这次会议改变了你的一项依赖”
* “你要注意某个 deadline 被提前了”
* “这项决策会影响你正在做的功能”
* “你的 API 可能要调整”
* “你的合作方说了你需要处理的风险”
* “这是你必须看的 2 句话，其它你不用管”

这才是你要的“异步协作的终极形态”。
